# Creator Workflow Idea Framework

Version: v0.1  
Status: Hermes-confirmed initial idea framework  
Purpose: establish the complete initial demand framework before handing theory expansion to Codex

---

## 1. Why this document exists

This document is the bridge between the user's rough product idea and a bounded Codex long task.

The user has clarified a critical product direction:

```text
I am not a professional AI programming operator. I want the system to encapsulate scattered AI workflow capabilities so I can focus on creating ideas, adjusting broad direction, judging satisfaction, and asking for corrections based on the product result.
```

This cannot be handed to Codex as a vague request. First, Hermes must interrogate the demand, define the problem world, identify the core principles, and produce a complete initial framework. Only then should Codex receive a bounded work order to expand the framework into professional theory and application scenarios.

This document is the initial framework.

---

## 2. Surface ask

The user's surface ask is:

```text
Use the demand-interrogation front end to deeply question the idea from multiple angles, establish an initial idea/demand document framework, then transform it to the point where Codex can continue and improve it into full professional theory.
```

The user is not asking for immediate code, UI, or tool implementation.

The user is asking for:

```text
idea clarity -> demand framework -> Codex-ready theory task
```

---

## 3. Real objective

The real objective is:

```text
Create a creator-first AI workflow operating system where the user stays in the role of creator, direction owner, satisfaction judge, and correction requester, while Hermes, Codex, Loop, Verifier, Research, Knowledge Base, and Git/GitHub operate as encapsulated internal subsystems.
```

The project must prevent two failure modes:

1. **Engineering exposure failure**: the user is forced to operate internal mechanisms such as Codex work orders, loop state, verifier commands, research-pack layers, git diffs, or knowledge-base promotion rules.
2. **Chat simplification failure**: the system hides so much rigor that it becomes a vague chat assistant with no contracts, no evidence, no verifier gate, no durable memory, and no productization path.

The correct product direction is:

```text
simple creator-facing surface + rigorous internal operating system
```

## 5. Solution world

The solution is a two-layer product model.

### 5.1 User-facing layer

The user-facing loop is:

```text
idea -> direction check -> system execution -> artifact preview -> satisfaction feedback -> correction
```

This is the surface the user should operate.

The user can say:

```text
I have an idea.
This direction is right/wrong.
I am satisfied/not satisfied.
This is too shallow/deep/complex.
Continue.
Stop.
Fix the direction.
Fix the details.
```

### 5.2 Internal operating layer

The internal system loop is:

```text
Hermes intake -> demand interrogation -> research if needed -> Demand Contract
-> .loop state -> Codex bounded WORK_ORDER -> execution
-> Verifier evidence gate -> stop state -> feedback routing
-> Knowledge Base / workflow learnback
```

The user should not need to manually operate this layer.

## 6. Core principles to preserve

This framework inherits the ten core principles:

1. **Creator Stays Creator** — the user stays in idea/direction/satisfaction/correction roles.
2. **Encapsulation Preserves Rigor** — hiding complexity must not remove internal checks.
3. **Demand Before Execution** — ambiguous ideas must become executable demand before Codex acts.
4. **Bounded Executor, Not Product Owner** — Codex executes but does not decide product direction.
5. **Verification Is Independent Evidence** — Codex self-report is not completion.
6. **State Must Outlive Chat** — long-task state lives in durable artifacts.
7. **Knowledge Promotion Has Gates** — raw evidence cannot become stable knowledge directly.
8. **HumanGate Protects Authority** — the user decides direction, risk, value, scope, satisfaction, and authorization.
9. **Satisfaction Is Product Evidence** — user satisfaction changes internal routing.
10. **Theory Must Drive Productization** — theory must affect future behavior, interfaces, work orders, or verifier gates.

These principles should be treated as acceptance criteria for the next Codex theory round.

## 7. Demand interrogation dimensions

Before Codex expands theory, the following dimensions must be preserved.

### 7.1 User role dimension

Question:

```text
What does the user actually own?
```

Answer:

```text
idea, direction, value judgment, satisfaction, risk boundary, major correction, stop/continue decision
```

The user does not own internal execution mechanics.

### 7.2 Encapsulation dimension

Question:

```text
What must be hidden from the user by default?
```

Answer:

```text
Codex details, loop files, verifier implementation, research layers, git mechanics, work order syntax, knowledge promotion internals
```

But evidence summaries and risk signals must remain visible.

### 7.3 Execution dimension

Question:

```text
How does the system execute without user micromanagement?
```

Answer:

```text
Hermes compiles demand and work orders; Codex executes bounded tasks; Loop preserves state; Verifier checks evidence; Hermes reports product status and asks only meaningful HumanGate questions.
```

### 7.4 Feedback dimension

Question:

```text
How does the user's satisfaction feedback affect the system?
```

Answer:

```text
Feedback routes to Done, Repair, revised Demand Contract, HumanGate, continuation, stop, Knowledge Base learnback, or workflow patch candidate.
```

### 7.5 Productization dimension

Question:

```text
How does theory become actual product behavior?
```

Answer:

```text
Each theory claim must map to a user-facing scenario, internal system action, verifier signal, and future product/interface behavior.
```

## 8. Initial product hypothesis

The product hypothesis is:

```text
A non-professional AI programming user can operate a sophisticated AI workflow operating system if the product surface is creator-first and the internal system preserves rigorous demand, execution, verification, state, and knowledge controls.
```

The expected product experience is:

```text
User: I have an idea.
Hermes: I understand the direction like this. Confirm or adjust?
User: Confirm / adjust.
System: executes internally through Research / Codex / Loop / Verifier.
Hermes: Here is the artifact preview, evidence summary, risk, and feedback options.
User: satisfied / not satisfied / direction wrong / detail wrong / continue / stop.
System: routes feedback into repair, new demand contract, HumanGate, or learnback.
```
